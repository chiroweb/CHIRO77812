# 치로웹디자인 블로그 글쓰기 가이드

> 이 가이드는 치로웹디자인 블로그의 모든 글에 적용됩니다.
> AI가 글을 작성할 때 반드시 이 가이드를 먼저 읽고 따라야 합니다.

---

## 0. 독자 페르소나

**이름:** 김 대표 (가상)
**나이:** 35~50세
**역할:** 소규모 비즈니스 의사결정권자 (대표, 팀장, 마케팅 담당)
**기술 수준:** 비개발자. HTML이 뭔지는 알지만 코드를 쓰지는 않음
**목표:** "우리 회사 홈페이지, 이대로 괜찮은 건가?" 에 대한 답을 찾고 있음
**읽는 방식:** 모바일에서 스크롤. 첫 3문장에서 읽을 가치 판단. 표와 리스트 선호
**핵심 니즈:** 깊은 정보를 원하지만, 전문 용어만으로 설명되면 이탈

**원칙:** 아무것도 몰라도 이해할 수 있을 정도로 풀어쓰되, 전문가가 읽어도 가볍지 않은 깊이.

## 1. 절대 원칙 (이거 어기면 다시 쓴다)

### 1-1. 첫 문장에서 결론부터
```
❌ "오늘날 디지털 마케팅 환경이 빠르게 변화하고 있습니다."
✅ "2026년, 검색 트래픽의 25%가 AI로 이동했습니다."
```
독자는 결론을 먼저 보고, 근거를 읽을지 결정한다.

### 1-2. AI가 자주 쓰는 표현 절대 금지
이 표현이 나오면 해당 문장을 삭제하고 다시 쓴다:
- "결론적으로", "요약하자면", "종합하면"
- "여러분", "독자 여러분"
- "~에 대해 알아보겠습니다", "~를 살펴보겠습니다"
- "~하는 것이 중요합니다", "~해야 합니다"
- "다양한", "효과적인", "혁신적인", "최적의", "완벽한"
- "~의 세계로 떠나보겠습니다"
- "~라고 할 수 있습니다"
- "이처럼", "이렇듯", "이와 같이"
- "깊이 있는", "심도 있는" (메타 표현 — 깊이를 보여주지 말고 실제로 깊이를 제공하라)

### 1-3. 추상어 금지 — 구체어로 교체
```
❌ "빠른 제작 기간"
✅ "평균 2주. Startup 플랜은 3~5일."

❌ "높은 검색 노출 효과"
✅ "구글 리치 결과 반영 2~6주, AI 검색 인용 증가 3~8주."

❌ "합리적인 가격"
✅ "175만원부터. 하루 약 4,800원."
```

### 1-4. 한 단락 = 한 주장
한 단락에 두 가지 이야기를 섞지 않는다. 주장이 바뀌면 단락을 나눈다.

### 1-5. 마침표로 끝낸다
느낌표(!) 사용 금지. 물음표는 질문일 때만.

## 2. 글의 구조 템플릿

### 도입부 (3문장 룰)
1. **팩트 문장:** 숫자 또는 사실로 시작. "2026년 현재, ~"
2. **문제 문장:** 이 팩트가 독자에게 왜 문제인지. "당신의 사이트가 ~하지 않다면, ~을 놓치고 있는 것입니다."
3. **약속 문장:** 이 글에서 무엇을 얻을 수 있는지. "이 글에서 ~의 차이와 ~하는 방법을 정리합니다."

### 본문 (B-E-R 구조)
각 섹션마다:
- **B (Brief):** 핵심 주장 1문장
- **E (Evidence):** 근거, 데이터, 사례, 비교표
- **R (Result):** "그래서 뭐가 달라지는가" — 독자에게 미치는 영향

### 마무리 (액션 트리거)
- 요약하지 않는다 (독자가 방금 읽었는데 왜 다시 말하나)
- 대신: 독자가 지금 당장 할 수 있는 행동 1가지를 제시
- 또는: 치로웹디자인의 관련 서비스로 자연스럽게 연결

## 3. "전문가처럼" 보이는 5가지 기법

### 3-1. 반대 관점 먼저 인정 후 반박
```
"SEO만으로도 충분하다고 생각할 수 있습니다. 
실제로 전체 검색 트래픽의 75%는 여전히 전통 검색 엔진에서 옵니다. 
하지만 나머지 25%를 무시하면, 매달 잠재 고객 4명 중 1명을 경쟁사에 넘기는 것과 같습니다."
```

### 3-2. 구체 숫자·날짜·출처
```
❌ "많은 사람들이 AI 검색을 사용합니다."
✅ "ChatGPT 월간 활성 사용자 3억 명 (2026년 1분기 기준)."
```

### 3-3. "왜 그런가"의 메커니즘 설명
```
❌ "구조화 데이터가 SEO에 좋습니다."
✅ "구조화 데이터(JSON-LD)는 검색엔진이 페이지 내용을 '이해'하게 만드는 코드입니다. 
    구글은 이 코드를 읽고 검색 결과에 별점, FAQ, 가격 같은 리치 결과를 표시합니다. 
    리치 결과가 표시되면 클릭률이 평균 30% 올라갑니다."
```

### 3-4. 기존 통념 도전
```
"대부분의 웹 에이전시는 '예쁜 홈페이지'를 만듭니다. 
하지만 예쁜 홈페이지와 일하는 홈페이지는 다릅니다."
```

### 3-5. 비대칭 인사이트
독자가 "이건 몰랐는데"라고 느끼는 정보. 이미 다 아는 이야기를 반복하지 않는다.
```
❌ "모바일 최적화가 중요합니다." (누구나 안다)
✅ "아임웹으로 만든 사이트의 평균 Lighthouse 점수는 50~70점대입니다. 
    구글은 Core Web Vitals 점수가 낮은 사이트의 검색 순위를 직접적으로 낮춥니다."
```

## 4. SEO·AEO·GEO 통합 원칙

### 4-1. title·H1·첫 100자 키워드 법칙
- `title` (frontmatter): 핵심 키워드를 앞에 배치
- 첫 `##` (H2): 글의 핵심 질문을 키워드 포함하여 작성
- 첫 100자 안에 핵심 키워드 최소 1회 등장

### 4-2. FAQ 섹션 의무화
모든 글 하단에 "자주 묻는 질문" 섹션 포함 (2~4개).
- 실제로 독자가 궁금해할 질문
- 답변은 2~3문장으로 완결
- FAQPage JSON-LD 스키마 자동 생성 대상

### 4-3. 내부 링크 룰
- 글 본문에 치로웹디자인 관련 페이지 링크 최소 2개
- 자연스러운 맥락에서만. 억지로 넣지 않는다.
- 링크 텍스트는 "여기", "이곳" 금지. 설명적 텍스트 사용.

### 4-4. Schema 자동 생성용 frontmatter
```yaml
---
title: "글 제목"
slug: "url-slug"
excerpt: "2문장 요약 (meta description으로도 사용)"
category: "AEO/SEO" | "웹사이트 제작" | "홈페이지 관리법" | "마케팅"
publishedAt: "2026-MM-DD"
author: "최정원"
keywords: ["핵심키워드1", "핵심키워드2", "롱테일키워드"]
schemaType: "BlogPosting" | "HowTo" | "TechArticle"
---
```

## 5. 톤 가이드

### 치로 톤 ✅
- 짧은 문장. 한 문장에 하나의 메시지.
- 존댓말 (~합니다) 일관 사용.
- 단정적이되 근거 있는 톤. "~입니다." (추측이 아님)
- 기술적이되 인간적인. 전문 용어는 쓰되 바로 다음에 쉬운 설명.
- 브랜드명 직접 사용: "치로는~", "치로웹디자인은~"

```
✅ "치로는 빌더를 사용하지 않습니다. 모든 코드를 직접 작성합니다. 
    그래야만 가능한 구조가 있기 때문입니다."
```

### 피해야 할 톤 ❌
```
❌ 강의 톤: "자, 그러면 이제 SEO에 대해 알아볼까요?"
❌ 블로그 잡담: "안녕하세요~ 오늘은 AEO에 대해 이야기해보려고 해요!"
❌ 논문 톤: "본 연구에서는 AEO의 효과성을 검증하고자 한다."
❌ 과잉 겸손: "부족하지만 도움이 되셨으면 좋겠습니다."
❌ 과잉 자신감: "이 글만 읽으면 AEO 전문가가 됩니다!"
```

### 톤 비교표
| 상황 | ❌ 피하기 | ✅ 치로 톤 |
|------|---------|----------|
| 서두 | "오늘은 ~ 알아보겠습니다" | 팩트 문장으로 시작 |
| 설명 | "쉽게 말해서~" | 쉬운 설명을 바로 제공 (메타 표현 불필요) |
| 강조 | "매우 중요합니다!" | "~없으면 ~를 잃습니다." (결과로 강조) |
| 마무리 | "도움이 되셨길 바랍니다" | 행동 1가지 제시 또는 서비스 연결 |

## 6. 금지 표현 사전

| 금지 표현 | 이유 | 대체 |
|----------|------|------|
| "결론적으로" | AI 티가 남 | 삭제. 마지막 단락이 결론이니 표시 불필요 |
| "여러분" | 블로그 잡담 톤 | "당신의 사이트" 또는 주어 삭제 |
| "~하시면 좋습니다" | 모호한 권유 | "~해야 합니다" 또는 "~하지 않으면 ~를 잃습니다" |
| "다양한" | 구체성 제로 | 실제 항목을 나열 |
| "효과적인/혁신적인/최적의" | 의미 없는 수식어 | 삭제하고 구체적 결과로 대체 |
| "~에 대해 알아보겠습니다" | AI 도입부 패턴 | 팩트 문장으로 시작 |
| "~라고 할 수 있습니다" | 회피형 단정 | "~입니다." |
| 느낌표 (!) | 과잉 에너지 | 마침표 (.) |
| 이모지 | 비전문적 | 사용 금지 |
| "깊이 있는/심도 있는" | 메타 표현 | 실제 깊이를 보여주면 됨 |
| "활용하다" | 한국어 번역체 | "쓰다", "사용하다" |
| "진행하다" | 관료체 | 구체 동사 ("만들다", "설계하다", "적용하다") |

## 7. 글 작성 워크플로

### 단계
1. **주제 받기** — 사용자가 주제를 지정
2. **키워드 매핑** — 핵심 키워드 3~5개 선정 (frontmatter keywords)
3. **구조 작성** — H2 기준으로 섹션 목차 설계
4. **본문 작성** — 각 섹션 B-E-R 구조로 작성
5. **셀프 체크** — 아래 체크리스트 15개 통과 확인
6. **MDX 파일 생성** — frontmatter + 본문
7. **빌드 테스트** — 에러 없이 렌더 확인

### 셀프 체크리스트 (발행 전 필수)
- [ ] 첫 문장이 팩트 또는 결론인가
- [ ] 금지 표현 사전에 있는 표현이 없는가
- [ ] 추상어 대신 구체 숫자/사례를 썼는가
- [ ] 한 단락에 한 주장만 있는가
- [ ] 느낌표를 사용하지 않았는가
- [ ] "여러분"을 사용하지 않았는가
- [ ] H2 섹션이 최소 3개 있는가
- [ ] FAQ 섹션이 있는가 (질문 2~4개)
- [ ] 내부 링크가 최소 2개 있는가
- [ ] frontmatter keywords가 3~5개인가
- [ ] 첫 100자 안에 핵심 키워드가 있는가
- [ ] title에 핵심 키워드가 앞에 있는가
- [ ] excerpt가 2문장 이내인가
- [ ] 글 길이가 1500~2500자인가
- [ ] 2026년 기준 정보인가 (날짜가 오래된 데이터 없는가)

## 10. AI에게 전달할 명령 예시

### 기본 명령
```
BLOG_WRITING_GUIDE.md를 읽고, [주제]로 블로그 글 1개를 작성해줘.
src/content/blog/에 MDX 파일로 저장해.
```

### 상세 명령
```
BLOG_WRITING_GUIDE.md를 읽고, 다음 주제로 블로그 글을 작성해줘:

주제: [주제]
핵심 키워드: [키워드1], [키워드2]
목표 독자: 홈페이지 리모델링을 고민하는 중소기업 대표
글 길이: 2000자 내외
톤: 치로 톤 (가이드 섹션 5 참고)

src/content/blog/[slug].mdx 파일로 저장해.
```

---

## 글 사양 (고정)

- **길이:** 2000 ± 500자 (한국어 기준)
- **독자:** 깊은 정보를 원하는 의사결정권자. 기술 비전문가도 이해 가능하게
- **시점:** 항상 2026년 기준
- **깊이:** 표면적 정보 나열이 아닌, "왜 그런가"의 메커니즘까지 설명
- **형식:** MDX (frontmatter + 마크다운)
- **저장 위치:** `src/content/blog/[slug].mdx`
